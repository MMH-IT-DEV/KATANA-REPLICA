Architectural Specification and Replication Guide: Katana MRP Stock Module

1. Executive Overview and System Architecture

The Stock module within the Katana Manufacturing Resource Planning (MRP) ecosystem functions as the central nervous system for inventory operations, serving as the convergence point for procurement, production, sales, and financial valuation. For a development team tasked with replicating this system, it is imperative to understand that the "Stock" page is not merely a passive reporting interface but a dynamic, event-driven calculation engine that maintains the perpetual state of inventory across multiple dimensions. The system is designed to provide real-time visibility into the availability of raw materials and finished goods, handling the complex state transitions that occur between "in-stock," "committed," and "expected" inventory levels.1

The architectural backbone of the Stock module relies on a variant-centric data model. While products and materials are categorized conceptually, the atomic unit of inventory tracking is the Variant. Every quantitative calculation—whether for stock availability, valuation, or batch tracking—is keyed to a specific variant ID within the context of a specific Location ID.3 This multi-location dependency means that the replica must process all inventory queries with a composite key of {variant_id, location_id}, ensuring that stock levels in a "Main Warehouse" are legally and logically distinct from those in a "Retail Outlet" or "Shop Floor."

Replicating the Katana Stock module requires the construction of five distinct but interconnected functional sub-modules, which correspond to the primary tabs in the user interface: Inventory, Batches, Stock Adjustments, Stock Transfers, and Stocktakes.5 Underpinning these tabs is a shared state machine that enforces data integrity. For instance, a "Stock Transfer" in the Received state effectively functions as a simultaneous debit and credit to the inventory ledgers of two different locations, while a "Stock Adjustment" acts as a unilateral modification to the ledger with direct financial implications.6

The system architecture must also account for high-concurrency transactional updates. In a live manufacturing environment, a sales order might commit stock at the exact same millisecond that a manufacturing order consumes it. The replica must implement database transaction isolation levels that prevent race conditions, particularly when calculating the "Committed" column, which serves as the primary guardrail against overselling.3 Furthermore, the integration with external platforms such as Shopify, WooCommerce, and QuickBooks Online (QBO) imposes strict requirements on data latency; stock changes in the replica must trigger webhooks or API calls to synchronize external systems in near real-time.9

2. The Inventory Tab: The Master Calculation Engine

The Inventory tab is the default view and the most computationally intensive component of the Stock screen. It provides a real-time snapshot of item availability and acts as the primary interface for master planning. Replicating this screen requires implementing a robust calculation engine that updates four key inventory states instantaneously upon the modification of any related document (Sales Order, Purchase Order, Manufacturing Order).

2.1 The Four Pillars of Inventory Logic

The foundational logic of the Katana Stock screen rests on the mathematical relationship between physical stock presence and theoretical demand. The replica must track four primary variables for every variant-location pair, updated continuously via an event bus or trigger system.

2.1.1 In Stock: The Physical Reality

The In Stock column represents the quantity of items physically present in the warehouse.3 This value serves as the "source of truth" for physical audits and valuation.

  • Source of Data: This value is the summation of all finalized stock movements. It is calculated as:
    $$ \text{In Stock} = (\Sigma \text{Purchases Received} + \Sigma \text{Mfg Completed} + \Sigma \text{Adjustments} + \Sigma \text{Transfers In}) - (\Sigma \text{Shipments Dispatched} + \Sigma \text{Mfg Consumed} + \Sigma \text{Transfers Out}) $$
  • Immutability: A critical design constraint for the replica is that the "In Stock" value is never directly editable in the grid. It is a read-only derivative of the transaction ledger. Any attempt to modify this value must force the user through a "Stock Adjustment" workflow to ensure an audit trail exists.6
  • Batch Aggregation: If batch tracking is enabled for a variant, the "In Stock" value displayed in the main inventory list must be the aggregate sum of all valid batch quantities (including potentially expired batches if not filtered) for that variant.11

2.1.2 Committed: The Allocated Demand

The Committed column represents stock that is physically present (or expected) but contractually reserved for open orders. This is the most volatile column, as it reacts to order lifecycle changes.

  • Trigger Logic:
  • Sales Orders (SO): When an SO is created, the line items are immediately added to "Committed".3 The replica must distinguish between "Open" and "Completed" SOs; once an SO is fulfilled, the quantity moves from "Committed" to a deduction in "In Stock."
  • Manufacturing Orders (MO): Raw materials required for an Open or Work in Progress MO are added to "Committed." This ensures that ingredients sitting on the shelf are not accidentally sold or used for other jobs.3
  • Lifecycle Dependencies: The replica must implement a state listener for MOs.
  • Status: Not Started: Ingredients are Committed.
  • Status: Work in Progress: Ingredients remain Committed.
  • Status: Done: Ingredients are removed from Committed and deducted from In Stock.8
  • Visual Warning: If Committed > In Stock, the system technically allows the state (backorder situation), but the "Calculated Stock" will turn negative, signaling a shortage.

2.1.3 Expected: The Inbound Supply

The Expected column tracks inventory that has been ordered or scheduled but not yet received. This visibility prevents over-purchasing.

  • Trigger Logic:
  • Purchase Orders (PO): Items on open POs are "Expected".3
  • Manufacturing Orders (MO): The finished goods resulting from an open MO are "Expected".8
  • Stock Transfers: Items in the "In Transit" state are "Expected" at the destination location.12
  • Partial Reception Logic: The replica must handle partial receptions. If a PO for 100 units has 40 units received, the "Expected" column must drop to 60, while "In Stock" increases by 40. This requires the "Expected" value to be calculated as Order Quantity - Received Quantity.3

2.1.4 Calculated Stock: The Master Planning Metric

The distinct differentiator for Katana’s logic—and the most critical formula for the replica—is the Calculated Stock metric. This derived value serves as the primary indicator for purchasing decisions.

The Formula:

$$\text{Calculated Stock} = \text{In Stock} - \text{Committed} + \text{Expected} - \text{Safety Stock}$$

Architectural Implications:

  • Purpose: This formula projects future availability. By including "Expected," it prevents the user from ordering stock that is already inbound. By subtracting "Committed," it ensures the user doesn't rely on stock that is already spoken for.3
  • Safety Stock Role: The Safety Stock level is a user-defined threshold.3 It acts as a customized buffer for each variant. If the Calculated Stock is negative, it implies the projected inventory will fall below the safety threshold, triggering a replenishment alert.13

2.2 Visual Intelligence: Missing/Excess Indicators

To assist users in rapid decision-making, the replica's UI must translate the mathematical result of the Calculated Stock formula into a color-coded visual indicator, referred to in the research as Missing/Excess.

  • Logic Definition:
    $$ \text{Missing/Excess} = (\text{In Stock} - \text{Committed} + \text{Expected}) - \text{Reorder Point} $$
    (Note: The terms Reorder Point and Safety Stock are functionally linked in this context. The "Missing" value tells the user exactly how far below the safety threshold they are.14)
  • UI State Requirements:
  • Red (Negative Value): Indicates a deficit. The calculated stock is below the safety stock level. The UI must vividly highlight this row to prompt immediate action.14
  • Neutral/Green (Positive Value): Indicates a healthy surplus.
  • Sorting: Users must be able to sort the grid by this column to bring all "Missing" items to the top, facilitating a prioritized replenishment workflow.

2.3 Smart Replenishment: The "Make" and "Buy" Workflows

Replicating the functionality of the Stock page requires implementing the "Quick Add" buttons (represented as a + or specific Make/Buy icons) located at the end of each inventory row. These buttons are context-aware and initiate the replenishment workflow.5

Operational Logic:

  • Context Awareness: The buttons must dynamically render based on the item type and location permissions.
  • Product Variant: Shows a "Make" button (triggers Manufacturing Order).
  • Material Variant: Shows a "Buy" button (triggers Purchase Order).
  • Permissions: If the current location does not have "Make" capabilities (e.g., a retail store location vs. a factory location), the "Make" button must be disabled or hidden.5
  • Auto-Fill Logic (The "Smart" Aspect): When a user clicks the replenishment button on a row with a negative Calculated Stock:
  • The system opens a creation modal (PO or MO).
  • Critical Requirement: The quantity field in this modal should pre-fill with the absolute value of the deficit to bring the Calculated Stock back to zero (or the safety stock level). While the documentation describes this as a "quick way to re-establish optimal levels," a high-fidelity replica should calculate this delta automatically: Replenishment Qty = abs(Calculated Stock).5

2.4 Valuation and Costing Columns

The Inventory tab acts as the primary interface for financial inventory valuation. The replica must maintain a real-time weighted average cost engine.

  • Moving Average Cost (MAC): Katana utilizes a MAC method. The "Cost per unit" displayed is a calculated average, not a static field.
  • Formula: $\frac{\text{Total Value of Stock}}{\text{Total Quantity In Stock}}$
  • Update Trigger: Every time a Purchase Order is received, the MAC must be recalculated.
  • $$ \text{New MAC} = \frac{(\text{Current Qty} \times \text{Current MAC}) + (\text{Received Qty} \times \text{Purchase Price})}{\text{Current Qty} + \text{Received Qty}} $$
  • Total Value: Calculated as In Stock Quantity * Average Cost.
  • Currency Handling: In multi-currency environments, the Inventory view typically normalizes values to the base currency of the tenant.16

3. Batch Tracking Sub-System: Traceability Architecture

The Batches tab introduces a layer of complexity over the standard inventory count. For industries like food, beverage, and pharma, inventory is not fungible; it is distinct based on production lots.1 The replica must implement a sub-ledger that tracks quantities at the {variant_id, batch_id, location_id} level.

3.1 Batch Data Model and Schema

To support the functionality described in the research, the database schema must support a one-to-many relationship between Variant and Batch.

  • Required Fields:
  • batch_number (String): The human-readable identifier (e.g., "LOT-2023-X").
  • expiration_date (Date): Critical for the FEFO (First Expired, First Out) logic.11
  • created_at (Timestamp): Used for sorting and FIFO logic.
  • barcode (String): Specific barcode for the batch to facilitate scanning operations.17
  • API Sorting: The Katana API returns batches sorted by location_id ASC, variant_id ASC, and batch_id DESC.17 This implies the UI should default to showing the most recent batches first within a location grouping.

3.2 Visibility Logic and The "Empty Batches" Toggle

A specific requirement for the Batches tab is the handling of zero-quantity batches.

  • Default State: The system filters out batches where quantity == 0 to reduce clutter.
  • Toggle Requirement: A boolean toggle "Show empty batches" must be implemented. When enabled, the query filter changes to include batches with quantity = 0. This is essential for traceability—users often need to look up an old, depleted batch to see which customers received it during a recall scenario.11
  • Expiration Filtering: Similarly, expired batches are often hidden or visually flagged. The replica should implement a visual state (e.g., red text or a "strikethrough") for batches where expiration_date < current_date.

3.3 Batch Assignment in Transactions

When stock is "Committed" or sold, the replica must handle the assignment of specific batches.

  • Allocation Logic: Unlike the general "Committed" bucket in the Inventory tab, the Batch logic requires specific assignment.
  • Manual Assignment: The user manually selects "Batch A" on the Sales Order.
  • Automatic Assignment: The system can suggest batches based on FEFO (First Expired First Out).
  • Consistency Check: The system must enforce that the sum of quantities across all batches for a specific variant equals the total "In Stock" figure for that variant in the main Inventory tab. Any discrepancy here indicates a data integrity failure (phantom inventory).

4. Stock Adjustments: The Correction Engine

The Stock Adjustments tab manages manual interventions. This is the mechanism for handling shrinkage, theft, damage, or initial stock loads. It is distinct from the other tabs because it represents a unilateral change to the ledger, rather than a bilateral exchange (like a transfer or purchase).

4.1 Adjustment Object Structure

A Stock Adjustment is a transactional document, not just a field edit. The replica must treat it as an immutable record once confirmed.

  • Header Level Fields:
  • stock_adjustment_number: Auto-generated string (e.g., "SA-101"), but must be editable by the user.18
  • reason: A free-text description field (e.g., "Annual Count Discrepancy", "Damaged in Warehouse"). Note: Research indicates this is a text field, not a strict enum dropdown, allowing for flexible descriptions.18
  • created_at: Timestamp.
  • location_id: The warehouse where the adjustment applies.
  • Line Item Level Fields:
  • item_id: The variant being adjusted.
  • quantity: The adjustment amount (delta). A positive value increases stock; a negative value decreases it. The replica should calculate the "New Stock Level" for display, but store the delta.18
  • cost_per_unit: Defaults to the current average cost. Crucially, this is editable. If a user finds an item they thought was lost, they might re-enter it at $0 cost or the original cost. This allows for value corrections without quantity changes.16

4.2 Financial Integration and GL Impact

Building a replica requires strict adherence to accounting principles, as the Stock Adjustment tab is a primary source of financial journal entries.

  • Value Change Calculation:

    $$\text{Inventory Value Change} = \text{Adjustment Quantity} \times \text{Cost per Unit}$$
  • Average Cost Recalculation: If the cost_per_unit entered in the adjustment differs from the current weighted average cost, the system must recalculate the new average cost for the item immediately upon confirmation.16
  • QuickBooks/Xero Mapping: For users connected to accounting software, the replica must trigger a Journal Entry.19
  • Negative Adjustment (Shrinkage): Credit Inventory Asset, Debit Inventory Shrinkage/Expense.
  • Positive Adjustment (Found Stock): Debit Inventory Asset, Credit Inventory Shrinkage/Expense.
  • Account Mapping: The system needs a settings configuration where the user maps the specific General Ledger (GL) account for "Inventory Shrinkage".20

5. Stock Transfers: Multi-Location Logistics

The Stock Transfers tab manages the movement of goods between defined Locations (e.g., "Main Warehouse" to "Retail Store"). This tab introduces a "state machine" logic that differs from immediate adjustments, designed to handle the time gap where inventory is physically moving but legally owned.

5.1 Transfer Lifecycle States

The system enforces a strict state flow to manage inventory ownership during transport. The replica must implement these three states with rigid validation.7

Inventory Impact (Destination)

5.2 The "In Transit" Void

A critical replication challenge is the handling of stock while "In Transit."

  • Asset Ownership: Legally and financially, the stock belongs to the company (typically the receiving location in internal transfers).
  • Visibility: The replica must visualize this stock. It is technically removed from the Origin's "In Stock" but has not yet hit the Destination's "In Stock." The "Expected" column at the destination acts as the placeholder for this inventory.12
  • Partial Availability Validation: During the "Created" phase, the system must validate availability. The interface displays "In Stock," "Expected," or "Not Available" for the line items. The replica must prevent the user from clicking "Sent" (moving to In Transit) if the Origin location lacks sufficient "In Stock" quantity to cover the transfer.12

5.3 Batch Preservation

Transfers must support batch preservation logic.

  • Identity Preservation: When Batch X expires on Jan 1st and is moved from Warehouse A to Warehouse B, it must arrive at Warehouse B as Batch X with the expiry Jan 1st. The system must not create a new batch ID; it must extend the existing batch record to the new location.21
  • Selection: The user must specify which batches are being moved at the point of sending.

6. Stocktakes: Periodic Reconciliation

The Stocktakes tab is designed for cycle counting and physical inventory verification. It is a workflow container that ultimately generates Stock Adjustments.

6.1 The Stocktake Lifecycle

The workflow moves through four distinct statuses, which the replica must enforce via the API and UI.22

  1. Not Started: The stocktake document is created. The user defines the Scope (e.g., "All materials at Location A" or "Category: Electronics").
  2. In Progress: Counting has begun. Users (or API inputs) are entering "Counted Quantities."
  3. Counted: Data entry is complete. Managers review discrepancies.
  4. Completed: The stocktake is finalized.
  • Action: This state transition triggers the automatic creation of a Stock Adjustment transaction.23
  • Logic: For every item where Counted Quantity!= Theoretical Quantity, a Stock Adjustment line item is generated for the difference.

6.2 The Snapshot Mechanism

To ensure accuracy, the replica must implement a "snapshot" mechanism.

  • Frozen Theoreticals: When a Stocktake is generated (or potentially when started, depending on exact configuration), the system must record the Theoretical Quantity (In Stock) at that exact timestamp.
  • Why? If operations continue (sales/shipping) while counting happens, the comparison must be against the frozen number. If the system compares against the live number at the moment of completion, the discrepancy calculation will be flawed due to movements that occurred during the count.

7. Technical Implementation Guide and API Structures

For a developer building this replica using Cursor AI, understanding the underlying API structure provides the clearest blueprint for the backend schema.

7.1 API Endpoint Architecture

Based on the research snippets, the backend relies on standard RESTful endpoints with specific filtering capabilities.

7.1.1 Inventory Endpoint (/v1/inventory)

  • Method: GET
  • Response Structure: List of inventory objects.
  • Rate Limiting: The replica must implement rate limiting headers (X-RateLimit-Limit, X-RateLimit-Remaining) to manage load, replicating Katana's default of 30 requests/second.24

 "updated_at": "2023-10-27T10:00:00Z"
}

7.1.2 Batch Stock Endpoint (/v1/batch_stocks)

  • Sorting Logic: location_id ASC, variant_id ASC, batch_id DESC. This specific sort order ensures that when a user looks at a list, they see batches grouped by location, then product, then newest first.17
  • Filtering: include_empty=true/false is a mandatory query parameter to support the UI toggle.17

7.1.3 Stock Adjustment Endpoint (/v1/stock_adjustments)

  • Filtering: Supports filtering by stock_adjustment_number, location_id, and date_range.25
  • Soft Delete: The API likely prevents hard deletion of adjustments that have been synced to accounting, suggesting a deleted_at column or a status="void" logic (include_deleted param).25

7.2 Webhooks and Event-Driven Updates

To replicate the "Real-Time" nature of Katana, the system cannot rely solely on polling. The replica must implement a webhook system.

  • Triggers: current_inventory.product_updated, current_inventory.product_out_of_stock.9
  • Usage: These webhooks should fire whenever the Calculated Stock crosses the Safety Stock threshold, triggering external notifications or automated workflows (e.g., via Make/Zapier).

8. Database Schema Recommendations

To support the features described, the following relational database schema is recommended for the replica.

Table: inventory_ledgers

  • id (PK)
  • variant_id (FK)
  • location_id (FK)
  • quantity_in_stock (Decimal)
  • quantity_committed (Decimal)
  • quantity_expected (Decimal)
  • reorder_point (Decimal)
  • average_cost (Decimal)
  • updated_at (Timestamp)
  • Index: Composite index on (variant_id, location_id) for O(1) retrieval speed.

Table: stock_movements (Audit Trail)

  • id (PK)
  • variant_id (FK)
  • location_id (FK)
  • batch_id (FK, Nullable)
  • quantity_change (Decimal)
  • movement_type (Enum: ADJUSTMENT, TRANSFER_IN, TRANSFER_OUT, SALES_DEDUCTION, PURCHASE_RECEIPT, MFG_CONSUMPTION, MFG_OUTPUT)
  • reference_document_id (FK to SO, PO, MO, SA)
  • cost_basis (Decimal)
  • created_at (Timestamp)

Table: batches

  • id (PK)
  • variant_id (FK)
  • batch_number (String)
  • expiration_date (Date)
  • barcode (String)
  • created_at (Timestamp)

9. Conclusion

Replicating the Katana MRP Stock page involves building a high-concurrency, transactional inventory engine. The complexity lies not in the visual display of the data, but in the interdependence of the columns and the strict state management of the workflows. The Calculated Stock formula is the heart of the system, governing the logic for purchasing and manufacturing triggers. The replica must treat every stock movement as a financial event, maintaining a rigid audit trail through Stock Adjustments and Transfers. By strictly adhering to the state definitions for "Committed" and "Expected" inventory, and ensuring that batch tracking is integrated into the core ledger rather than treated as an afterthought, the development team can successfully reconstruct the operational capabilities of the Katana platform.
