Architectural Specification and Implementation Guide for Katana MRP Items Module Replica

1. Executive Summary: The Centrality of the Items Module

In the ecosystem of Manufacturing Resource Planning (MRP), the "Items" module functions as the foundational registry and logic engine for the entire enterprise. For a developer tasked with replicating the Katana MRP platform using Cursor AI, it is imperative to understand that the Items page is not merely a static data entry form; it is the "Single Source of Truth" (SSOT) that governs the behavior of inventory management, production scheduling, sales execution, and financial cost accounting.

The architectural integrity of the Items module dictates the success of downstream processes. Every Sales Order (SO) relies on the definition of sellable products established here. Every Manufacturing Order (MO) queries the "Product Recipe" (Bill of Materials) and "Production Operations" (Routing) configurations hosted within this module to determine material requirements and capacity needs. Furthermore, the financial accuracy of the system—specifically the calculation of Cost of Goods Sold (COGS) and inventory valuation—is rooted in the costing logic defined at the item level.

The system employs a rigid yet polymorphic taxonomy, classifying entities into Products, Materials, and Services. This tripartite distinction is the primary architectural driver, influencing database schema design, API endpoint logic, and user interface state management.1 The replica must faithfully implement these distinctions, ensuring that "Materials" cannot be manufactured, "Services" do not track stock, and "Products" integrate the complex logic of multi-level Bills of Materials (BOMs).

This comprehensive report provides a detailed technical specification for the Items module, synthesizing data from Katana’s documentation, API references, and functional behaviors. It is designed to guide the backend schema design, frontend component architecture, and business logic implementation required to build a high-fidelity replica.

2. Entity Taxonomy and Data Modeling

The first and most critical architectural decision in building the Katana replica is the definition of the base classes for inventory items. Katana does not treat all items equally; it segregates them into three distinct types with unique behaviors, validation rules, and API interactions.

2.1. The Product Entity (Product)

Products represent the "Finished Goods" or "Subassemblies" within the manufacturing context. They are the output of production processes and are typically the primary revenue drivers for the business.

  • Definition & Role: A Product is an item that is manufactured or assembled. It serves as the parent entity for the "Make" workflow.1
  • Inventory Behavior: Products are fully stocked entities. The system tracks:
  • In Stock: Physical quantity on hand.
  • Expected: Quantity incoming from open Manufacturing Orders (MOs).
  • Committed: Quantity reserved for open Sales Orders (SOs).
  • Architectural Features:
  • Recipe Support: Only Products can possess a Bill of Materials (Product Recipe).
  • Routing Support: Only Products can have Production Operations (Labor/Machine steps).
  • Subassembly Logic: A Product can function as a component inside another Product’s recipe, facilitating multi-level BOM structures.3
  • API Identity: Products are accessed via the /v1/products endpoint. They carry specific boolean flags like is_producible and is_sellable that drive their behavior.4

2.2. The Material Entity (Material)

Materials represent "Raw Inputs" or "Components" acquired from external suppliers. They form the base layer of the manufacturing pyramid.

  • Definition & Role: A Material is a resource consumed during production. It cannot be manufactured within the system (unless converted to a Product type).1
  • Inventory Behavior: Like Products, Materials are tracked for In Stock, Expected (via Purchase Orders), and Committed (to Manufacturing Orders).
  • Architectural Constraints:
  • No Recipe/BOM: Materials do not have ingredients.
  • No Operations: Materials do not have routing steps.
  • Procurement Focus: Materials are tightly coupled with Supplier logic, Default Purchase Prices, and Purchase Units of Measure (UoM).5
  • API Identity: Materials are accessed via the /v1/materials endpoint. The is_producible flag is inherently false or non-existent for these entities.6

2.3. The Service Entity (Service)

Services represent intangible items that carry value but do not require physical storage or tracking.

  • Definition & Role: Services encompass labor charges, shipping fees, consulting hours, or overhead costs added to Sales Orders.
  • Inventory Behavior: Services are non-stocked. They effectively have infinite availability and do not trigger stock movements, commitments, or reorder alerts.2
  • Architectural Features:
  • Cost Accounting: Services have a cost that contributes to the margin calculation on a Sales Order but are not treated as inventory assets on the balance sheet.
  • Exclusion Logic: Services must be programmatically excluded from Picking Lists, Packing Slips, and Shipping integrations to prevent warehouse errors.
  • API Identity: Services likely share the generic Item structure or have a dedicated /v1/services endpoint, characterized by the absence of stock tracking fields.7

3. General Information Architecture: The Item Card

The "Item Card" serves as the primary user interface for defining the static attributes of an entity. In a replica built with Cursor AI, this form requires sophisticated state management to handle the interdependencies between identification fields, usability toggles, and unit conversions.

3.1. Identification and Categorization

The integrity of these fields is paramount as they propagate to every other module (Sell, Make, Buy, Stock).9

3.1.1. Item Name

The name field is the primary string identifier. While the system allows flexibility, the implementation should encourage specific naming conventions.

  • Naming Strategy: Best practices suggest generic parent names (e.g., "Dining Table") rather than attribute-laden names (e.g., "Dining Table - Oak - Large") when the Variant system is active. The system should ideally separate the "Base Name" from the "Variant Descriptors" to maintain clean data structures.8
  • Search Indexing: The name field must be indexed for high-performance full-text search across the application, as it is the primary lookup key in Sales Orders and MOs.

3.1.2. Category Management

The category field is not merely a text string but a relational link to a centralized Categories table.

  • Functionality: Categories group items for filtering in the "Items" list and for aggregated reporting in the "Insights" module.
  • Management Logic: The replica must implement a "Create on the Fly" mechanism within the Item Card, as well as a dedicated management view in Settings to Edit/Delete categories.
  • API Structure: The product object includes a category_name or category_id. If category_name is sent via API and does not exist, the system logic typically auto-creates it.4

3.1.3. Unit of Measure (UoM)

The UoM definition is critical for the mathematical consistency of the MRP engine.

  • Base UoM: This defines the unit for stockkeeping (e.g., pcs, kg, m). Once an item has associated stock transactions (MOs, POs, Adjustments), the implementation should strictly lock this field to prevent data corruption. Changing a UoM from pcs to kg post-transaction renders historical data meaningless.4
  • Purchase UoM & Conversion: For Materials, the system must support a purchase_uom distinct from the base uom.
  • Scenario: A business stocks "Wire" in meters (Base UoM) but buys it in spools (Purchase UoM).
  • Logic: The system requires a conversion_rate field (e.g., 1 Spool = 500 Meters). When a PO is created for 2 Spools, the inventory must increment by 1000 Meters.6

3.2. The Usability Logic Gates (Make/Buy/Sell)

Katana employs a unique "Usability" toggle system that dynamically alters the behavior of an item. These boolean flags—is_sellable, is_producible, is_purchasable—act as the primary state controllers for the UI.4

3.2.1. "I Sell This" (is_sellable)

  • Function: Marks the item as a finished good or merchandise.
  • UI Effect: Enables the "Default Sales Price" field.
  • System Effect: Adds the item to the query filter for the "Product" dropdown in Sales Orders. If disabled, the item cannot be added to a Quote or SO.4

3.2.2. "I Make This" (is_producible)

  • Function: Identifies the item as a manufactured product.
  • UI Effect: This is the master switch for the Product Recipe / BOM and Production Operations tabs. If unchecked, these tabs are hidden or disabled.
  • System Effect: Allows the item to be the target of a Manufacturing Order (MO).
  • Data Validation: Only items with this flag set to true can have rows in the recipes and product_operations tables.4

3.2.3. "I Buy This" (is_purchasable)

  • Function: Identifies the item as a procured good.
  • UI Effect: Enables "Default Purchase Price" and "Default Supplier" fields.
  • System Effect: Adds the item to the query filter for Purchase Orders (POs).
  • Hybrid Workflow (Outsourced Manufacturing): If a user checks both "Make" and "Buy," the system interprets this as an item that can be produced internally or outsourced to a contract manufacturer. This enables the "Outsourced Purchase Order" (OPO) workflow, where a PO is issued for the finished good, but raw materials are consumed from the user's inventory.3

3.3. Advanced Tracking: Batch and Serial Numbers

For regulated industries (food, pharma, electronics), the replica must support traceability.

  • Batch Tracking: A boolean toggle (batch_tracked). If enabled, the system enforces a strict validation rule: Every stock movement (IN or OUT) must be associated with a specific Batch ID and Expiration Date. This fundamentally changes the inventory ledger from a simple quantity counter to a batched ledger.8
  • Serial Tracking: Similar to batching but enforces a quantity of 1 per ID, ensuring individual unit traceability.

4. Polymorphism and Variant Management

Modern e-commerce and manufacturing require handling products that come in multiple variations (e.g., Size, Color, Material). Katana handles this through a robust "Parent-Child" variant architecture.

4.1. Variant Configuration Architecture

The Item Card allows users to define a matrix of options that generate specific Stock Keeping Units (SKUs).

  • Configuration Schema: The system supports up to 10 Variant Options (e.g., Color, Size, Style) per item.
  • Value Constraints: Each option can have up to 40 Option Values (e.g., Red, Blue, Green).13
  • Combinatorial Logic: The replica must implement a Cartesian product algorithm.
  • Input: Option A {Red, Blue} AND Option B {S, M}.
  • Output: 4 Variants: Red-S, Red-M, Blue-S, Blue-M.
  • Implementation Note: The "Generate Variants" function should be idempotent. If a user adds a third option "Green" later, the system should generate Green-S and Green-M without duplicating or deleting the existing Red/Blue variants, preserving their stock history.15

4.2. The Variants Grid

Located at the bottom of the Item Card, the Variants Grid allows for granular control over each child SKU.

  • Variant Code / SKU: This is the unique key for the variant. The system should offer an auto-generate feature (e.g., PARENT-OPT1-OPT2) but allow manual overrides.
  • Validation: The sku field must be unique across the entire variants table. Duplicates must be flagged immediately.16
  • Pricing Scopes (Global vs. Local):
  • Global: Users can set a "Default Sales Price" at the parent level.
  • Local: Users can override this price for specific variants (e.g., XL size costs $2.00 more). The database schema must support a sales_price on the Variant object that falls back to the Parent object if null.14

4.3. API Structure for Variants

The API treats variants as distinct objects linked to a product_id.

  • Endpoint: /v1/variants.
  • Attribute Payload: The config_attributes array holds the key-value pairs defining the variant (e.g., ``).17
  • Deletion Logic: When deleting a variant, the system must check for "Open Orders." If the variant is part of an active MO or SO, the deletion should be blocked or converted to an "Archival" state to maintain referential integrity.18

5. The Bill of Materials (Recipe) Subsystem

The Product Recipe / BOM tab is the heart of the MRP engine. It defines the dependency graph between a Product (Output) and its Materials/Subassemblies (Inputs).

5.1. Data Structure and Relationships

The BOM is not a simple list; it is a relational mapping table.

  • Parent Relationship: Links to a product_variant_id.
  • Child Relationship: Links to an ingredient_variant_id (which can be a Material OR another Product).19
  • Quantity: A float value defining the amount required to produce one unit of the parent.
  • Costing: The "Stock Cost" column displayed in the UI is a calculated field: Ingredient Average Cost * Quantity. It is read-only in this view because it is derived from the ingredient's current inventory valuation.20

5.2. Variant Mapping Logic

A sophisticated feature of Katana is "Variant Mapping," which the replica must implement to handle complex product lines without creating dozens of separate BOMs.

  • Scenario: A "Table" comes in "Oak" and "Pine."
  • Requirement: The "Oak Table" variant must consume "Oak Wood," and the "Pine Table" must consume "Pine Wood."
  • Implementation: The UI should allow the user to select the "Active Variant" context. The backend must store specific BOM rows that are conditional based on the parent variant ID. Alternatively, a "Smart Match" logic can be implemented where if the Ingredient has a "Color" option matching the Parent's "Color" option, it is automatically selected.21

5.3. Subassemblies and Recursive Explosion

The replica must support Subassemblies—products used as ingredients in other products.3

  • Recursive Logic: The MRP engine must implement a recursive "BOM Explosion" algorithm. To determine the raw material requirement for the top-level item, the system must traverse down the dependency tree (Parent -> Subassembly -> Raw Material).
  • Loop Detection: The system must validate against circular references. If Product A requires Product B, and Product B requires Product A, the system enters an infinite loop. The Item Card validation logic must detect and block these cycles upon saving the Recipe.20

6. Production Routing: The Operations Tab

While the BOM defines what is used, the Production Operations tab defines how it is made. This data drives the production scheduling and labor costing features.

6.1. Operation Steps Data Model

The Operations list is a sequential array of steps, critical for the "Shop Floor App" functionality.

  • Sequence: An integer defining the order of steps (1, 2, 3...).
  • Operation Name: A string describing the task (e.g., "Cutting", "Assembly").
  • Resource: A foreign key to the Resources table (Workstations or Operators). This link is vital for capacity planning.23
  • Cost Rate: A currency value (cost_per_hour). Defaults to the Resource’s standard rate but can be overridden per operation row.
  • Time per Unit: A duration field.
  • Calculation: Total Operation Cost = Time per Unit * Cost Rate.23

6.2. Capacity Planning Implications

The data entered here feeds the Master Production Schedule.

  • Logic: If an MO is created for 10 units, and the "Cutting" operation takes 30 minutes, the system books 300 minutes (5 hours) on the "Cutting Table" resource.
  • Replica Requirement: The Items module must expose this time data to the "Make" module's scheduling engine to visualize load vs. capacity.24

7. Cost Accounting Engine: The Valuation Logic

One of the most complex backend components of the Items page is the dynamic calculation of item costs. The Item Card displays these costs, but the logic is event-driven.

7.1. Moving Average Cost (MAC)

Katana uses MAC for inventory valuation.

  • Formula: New MAC = ((Current Qty * Current MAC) + (Incoming Qty * Incoming Price)) / (Current Qty + Incoming Qty).
  • Triggers: This calculation runs on every Purchase Order Receipt and Manufacturing Order Completion.
  • Replica Requirement: The Item Card's "Average Cost" field is read-only. The system must maintain a transaction ledger to recalculate this value in real-time. Manually editing cost is generally restricted to "Stock Adjustments".11

7.2. Product Cost Roll-up

For Products, the "Proxy Cost" (Estimated Cost) displayed on the Item Card is a summation:

  • Estimated Product Cost = Sum(Ingredient MAC * BOM Qty) + Sum(Operation Cost per Unit).
  • Dynamic Update: If the price of "Steel" (Material) rises due to a new PO, the Estimated Cost of the "Steel Table" (Product) must automatically update to reflect the new margin reality. This requires a dependency graph update mechanism in the backend.25

8. User Interface and Experience (UI/UX) Architecture

The Items Screen (List View) is the high-traffic control center. Replicating its UX requires attention to grid performance and state persistence.

8.1. Grid Functionality and State

  • Data Density: The grid must handle thousands of rows. Virtualization (rendering only visible rows) is recommended for the React/Vue frontend.
  • Sticky State: A critical UX requirement is "State Persistence." If a user filters for "Category: Furniture" and sorts by "Stock: Ascending," then navigates away and returns, the grid must restore this exact state. This implies saving view configurations in localStorage or User Preferences.27
  • Customizable Columns: Users can toggle visibility for columns like Image, SKU, Price, and Stock. The implementation should allow resizing and reordering columns.28

8.2. Bulk Actions Logic

To manage scale, the replica must implement "Bulk Actions" via multi-select checkboxes.29

  • Bulk Delete: Requires validation to ensure no selected items have open orders.
  • Bulk Edit: Allows modifying shared attributes (e.g., changing Category for 50 items at once).30
  • Export/Import Roundtrip: A common workflow is "Export to Excel" -> "Edit in Excel" -> "Re-import." The system must support this by generating CSVs with unique IDs (or SKUs) that allow the Importer to match and update existing records rather than creating duplicates.15

8.3. Custom Fields Architecture

Katana supports extended metadata via Custom Field Collections.

  • Schema: These are likely implemented as a JSONB column or an Entity-Attribute-Value (EAV) model in the database.
  • Configuration: Users define a "Collection" (e.g., "Tech Specs" containing "Voltage" and "Wattage") in Settings.
  • Assignment: The Item Card allows selecting a Collection, which dynamically renders the corresponding input fields in the UI. These fields must be accessible via API and visible on Sales Orders.32

9. API Architecture and Integration Points

For the replica to be extensible, it must mirror the Katana API structure. The Items module functionality is distributed across several RESTful endpoints.

9.1. Core Endpoints

  • GET /v1/products: Supports filtering by updated_at (for syncing), category, and sku. Returns the full object including nested variant summaries.4
  • GET /v1/materials: Similar structure but filtered for material-type entities.6
  • GET /v1/product_recipes: Returns the BOM rows. Critical for integrations that need to calculate raw material requirements externally.35
  • GET /v1/product_operation_rows: Returns routing steps. Used by external scheduling tools or custom Shop Floor apps.7

9.2. Webhooks

The replica should implement webhooks to notify external systems of changes.

  • Triggers: product.created, product.updated, product.deleted.
  • Payload: Should include the simplified object state to allow listeners (like eCommerce integrations) to update stock levels or prices immediately.7

10. Advanced Workflow Integrations

10.1. Kits and Bundles

The "Kit/Bundle" toggle activates a specific logic path: Auto-Assembly.

  • Logic: A Bundle is not "manufactured" via an MO. When a Bundle is sold on an SO, the system automatically deducts the stock of the constituent components.
  • Replica Requirement: The backend "Inventory Commitment" logic must detect is_auto_assembly = true and recursively commit the ingredients instead of the parent item.4

10.2. Missing/Excess Stock Logic

The Items grid often displays a "Missing/Excess" column, which is a calculated metric vital for planners.

  • Formula: Missing/Excess = (In Stock - Committed + Expected) - Reorder Point.
  • Visualization: If the result is negative, the UI highlights it in red, indicating an immediate need to reorder. This requires the grid to perform real-time math on multiple data columns.37

11. Implementation Roadmap for Cursor AI

To build this module effectively in Cursor AI, the following development sequence is recommended:

  1. Phase 1: Database Schema: Define items, variants, uoms, categories, and the polymorphic items table (distinguishing Product/Material).
  2. Phase 2: Base CRUD: Implement the Item Card with basic fields and the Make/Buy/Sell boolean logic.
  3. Phase 3: Variant Engine: Build the logic to generate variant rows from Option/Value inputs.
  4. Phase 4: BOM & Operations: Create the relational tables for Recipes and Routings, enforcing the "Product Only" constraint.
  5. Phase 5: Costing & Inventory: Implement the Moving Average Cost triggers and the "Missing/Excess" calculation formula.
  6. Phase 6: Bulk Actions & API: Build the CSV importer and the REST endpoints to expose the system to the outside world.

By adhering to this specification, the replica will not only look like Katana but will function with the same logical rigor, ensuring it can serve as a reliable ERP foundation.

12.1. The Variant Generator Algorithm

The variant generation logic is a critical UX component. It must be robust enough to handle updates without data loss.

  • Input Data: The user provides an array of Options, each with an array of Values.

  • Generation Step: The frontend calculates the Cartesian product: ,.
  • Merge Logic (Crucial):
  • Fetch existing variants for the item from the database.
  • Compare generated keys against existing keys.
  • Match: Do nothing (preserve existing ID, Stock, and Price).
  • New: Create a new variant record (Insert).
  • Missing: If a user removed a value (e.g., removed "Red"), mark the corresponding variant as archived rather than deleted if it has historical data. This "Soft Delete" strategy is essential for ERP audit trails.14

12.2. Recursive BOM Costing Algorithm

The "Estimated Cost" on a Product Card is not a static field; it is the result of a recursive calculation that must run efficiently.

  • Trigger: The calculation runs whenever the Product Card is loaded, or a background job updates it when ingredient costs change.
  • Algorithm:
    Python
    def calculate_product_cost(product_id):
       total_cost =     0
           # 1. Material Costs
       recipe_rows = get_recipe_rows(product_id)
           for row in recipe_rows:
           ingredient = get_item(row.ingredient_id)
               if ingredient.type == 'Material':
                   # Base Case: Use Moving Average Cost
               cost = ingredient.average_cost
               elif ingredient.type == 'Product':
                   # Recursive Case: Subassembly
               cost = calculate_product_cost(ingredient.    id)

           total_cost += (cost * row.quantity)

           # 2. Operation Costs
       op_rows = get_operation_rows(product_id)
           for op in op_rows:
           total_cost += (op.time_per_unit * op.cost_per_hour)

           return total_cost
  • Performance Optimization: For deep BOMs, this recursion can be slow. The replica should implement Memoization or store the "Last Calculated Cost" and only invalidate it when dependencies change (a "Dirty Flag" pattern).25

12.3. The "Missing Stock" Alert System

The Items list view provides a dashboard-like capability via the "Missing/Excess" column. This is the primary call to action for purchasing managers.

  • Data Sources:
  • In Stock: Current physical count (requires SELECT sum(qty) FROM stock_quants WHERE item_id = X).
  • Committed: Reserved for Sales/Manufacturing (requires aggregation of Open SO rows and Open MO ingredient rows).
  • Expected: Incoming (requires aggregation of Open PO rows and Open MO product rows).
  • Reorder Point: User-defined static threshold.
  • The Formula: Projected Availability = In Stock - Committed + Expected.
  • Alert Logic:
  • If Projected Availability < Reorder Point: Flag as Missing (Red).
  • Display Missing Qty = Reorder Point - Projected Availability.
  • This logic implies that the "Items" view performs heavy aggregation. For performance, these counters (committed, expected) are typically cached on the Item record and updated transactionally, rather than summed on every page load.37

12.4. Contract Manufacturing (OPO) Workflow Implementation

The "Outsourced Purchase Order" is a hybrid workflow that the Item Card settings (is_producible AND is_purchasable) must enable.

  • Configuration: The Item is marked as "Buy" (so it can be on a PO) and "Make" (so it has a BOM).
  • PO Creation Logic: When a user creates a PO for this item:
  1. The system identifies it as an OPO item.
  2. It prompts the user to select a "Track Ingredients" option.
  3. If selected, the system looks up the Product Recipe.
  4. It creates "Stock Movements" to consume the ingredients from the user's location (deducting raw materials) and add the finished good to the user's location upon receipt.
  5. The PO price represents the Service Fee paid to the manufacturer, not the full cost of the item.
  • Costing Implication: The final value of the item entering stock = PO Service Fee + Cost of Ingredients Consumed. This is a sophisticated "Landed Cost" calculation the replica must handle.3

13. Data Import/Export Architecture

Given the volume of data in ERPs, the "Items" page relies heavily on bulk operations. The replica needs a robust CSV parsing engine.

13.1. Import Template Design

Katana provides specific templates for "Add New Products" and "Update Existing Products."

  • Structure: The CSV acts as a flattened representation of the relational database.
  • Columns: Item Name, Category, UoM, Variant Code, Variant Option 1, Variant Value 1, Price.
  • Grouping Logic: The importer must be smart enough to group rows.
  • Scenario: The CSV contains 3 rows with Item Name "T-Shirt" but different Sizes (S, M, L).
  • Logic: The importer parses the first row to create the Parent Product. It then parses all 3 rows to create 3 Variant records linked to that Parent. It does not create 3 separate Products.15

13.2. Update vs. Create Logic

  • Key Identifier: The importer typically uses SKU (Variant Code) or Internal ID as the key.
  • Flow:
  1. Upload File.
  2. Validate Headers.
  3. For each row: Check if SKU exists.
  • Yes: Update fields (Price, Reorder Point).
  • No: Error (if in "Update" mode) or Create (if in "Add" mode).
  • Validation: The importer must validate referential integrity. If a row specifies Category: "NewCat", and "NewCat" doesn't exist, the system should either error out or auto-create the category based on user preference settings.31

14. Conclusion

The "Items" page in Katana MRP is a sophisticated intersection of inventory management, cost accounting, and production planning. For a developer replicating this system, the challenge lies not in the UI layout, but in the interdependencies of the data.

The system relies on a strict validation logic where the "Type" of item (Product/Material) dictates its behavior. The Variant system requires a robust implementation of attribute permutation and inheritance. The BOM and Operations modules are not merely lists but functional drivers that calculate costs and schedule capacity. Finally, the Usability Toggles (Make/Buy/Sell) act as the primary state controllers, enabling a polymorphic behavior where a single database entity can function as a raw material, a finished good, or an outsourced product depending on its configuration.

Successful replication requires a database schema that enforces referential integrity between Parents and Variants, and a frontend state machine that accurately reflects the constraints imposed by the "Make," "Buy," and "Sell" logic gates. The robustness of the Items module directly correlates to the accuracy of the entire ERP's reporting capability.
